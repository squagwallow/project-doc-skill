# project-doc-skill — Entry Document
Type: META | Bootstrap document. Read before every session.

## Purpose
Designing, building, and iterating a reusable LLM skill called build-new-project.skill that initializes a complete, LLM-readable external memory and documentation layer for any new project. The skill interviews the user, scopes the project, engineers all required prompts, and generates a complete repo ready to deploy. This project is simultaneously the first project running on its own protocol.

## Current Objective
Test skill v0.5 in Claude Code against a real project. Iterate based on results.

## Settled Decisions
- Thread-start prompt is the primary retrieval mechanism — LLM-agnostic.
- GitHub connector/direct repo access is the primary retrieval path when available.
- jsDelivr is a known fallback retrieval path, not the default.
- entry.md hard size limit is a performance requirement — context rot is real.
- Exit protocol equals manual compaction — model drafts, human reviews and commits.
- Settled decisions use "Defaults — Override With Cause" framing.
- Conditional reading must include explicit trigger conditions.
- Pre-task checklist belongs in both entry.md and thread-start prompt.
- AGENTS.md must be human-authored.
- Personal Context Layer is conditional required reading only — one central document.
- raw.githubusercontent.com is unverified for robots.txt — do not use until tested.
- Each project generated by the skill gets its own separate GitHub repo.
- The skill runs in Claude Code — generates a complete folder, not text blocks.
- Prompt engineer skill absorbed into prompt-engineer-entry.md per project.
- Tiers are examples not fixed categories — the skill is adaptive.
- Decomposition applies to all project types regardless of complexity.
- GitHub project connector is a viable retrieval path for Claude.ai sessions.
- Claude Code FILE:/DELETE:/COMMIT MESSAGE: format is the settled commit mechanism.
- Handoff log must include do-not-retry section and handoff completeness rules.
- Writing prompts must use two-pass draft pattern and evidence-based self-review.
- Style guides must use contrastive bad→good pairs for every prohibition.
- README.md is standard skill output generated at project creation.

## Defaults — Override With Cause

- Do not propose nesting generated projects inside this repo.
- Do not propose fixed template taxonomy — the skill is adaptive.

## Notes Block
Maintain a single, very compact THREAD NOTES block in every reply.

Update order (always before your main answer):
Prune: Remove items that are obsolete or irrelevant.
Merge: Combine related points into one short clause.
Tighten: Rewrite for maximum brevity and clarity.
Add: Only add new items important for future turns.

Format (exactly this structure, minimal lines):
THREAD NOTES (Turn N) ← replace N with the current turn number and increment by 1 each reply
✓ Settled: brief decisions, separated by · where possible
→ Active: current tasks / work in progress (max 3 items)
? Open: unresolved questions / forks (max 5 items)
Δ Meta: optional constraints/observations (only if truly needed; omit line entirely when empty)

Rules:
Always update THREAD NOTES first, then write your main reply.
Keep the whole block as short as possible while still accurate.
Prefer deleting and merging over letting it grow.
Do not restate full user messages; store only distilled essentials.
No emojis, bold, markdown headings, or bullet lists inside THREAD NOTES.
Use inline separators ( · ) instead of extra line breaks where you can.
When Turn N is greater than 20, be extra aggressive about pruning and consider suggesting a new thread if NOTES feel crowded.
